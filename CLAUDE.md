# CLAUDE.md — 토스증권 × Notion 주식 위젯

> Claude Code가 이 레포를 빌드할 때 따르는 지침. 상세 설계 근거는 [toss_notion_stock_widget_design.md](toss_notion_stock_widget_design.md), 프론트 UI 규칙은 [DESIGN.md](DESIGN.md)에 있다. 충돌 시 **설계서 > CLAUDE.md > 그 외**.

## 1. 목적 / 범위

폴링 기반 시세 위젯(시세·차트·ETF 히트맵)을 보여주고, 사용자의 **기존 Notion `종목 DB`에 `현재가`를 자동 주입**해 평가금액·평가손익·수익률(=Notion 수식)이 실시간에 가깝게 갱신되게 한다.

**성격**: 런타임 LLM이 없는 **결정론적 웹앱 + 데이터 동기화 파이프라인**. 여기서 "에이전트"는 *빌더로서의 Claude Code*만 가리킨다 — 런타임 서브에이전트/LLM 판단은 **없다**.

| 포함 | 제외 |
|------|------|
| 시세 위젯(테이블·캔들차트·ETF 히트맵·관심종목) | 자동 주문/매매(토스 Order 미사용, 조회 전용) |
| 토스 시세 → `종목 DB.현재가` 주기 동기화 | 런타임 LLM, 다중 사용자 SaaS |
| **(선택)** 토스 체결 → `매매일지 DB` 자동 임포트 | 평단가·평가손익 **재계산 후 Notion 쓰기**(Notion 수식 담당) |
| ETF 예상가 히트맵(구성종목 Notion 수동 등록) | 미래에셋·키움 등 **타 증권사 거래 자동 수집**(토스 API 밖 → 수동) |

## 2. 확정 사항 (이 레포의 전제)

- **레포**: 새 레포로 분리. npm workspaces 모노레포(`toss-notion-widget`).
- **스택**: TypeScript 전면 적용 / React + Vite(web) / Node + Express(server) / **Vitest**(테스트) / **zod**(스키마 검증).
- **백엔드 실행 환경**: **Railway 단일 Node+Express 서비스 + node-cron + SQLite(better-sqlite3)**. (대안 검토는 설계서 §6.)
- **토스 API 키 미수령**: 현재 **Mock/Fixture 우선** 개발. 토스 응답 고정값(fixture)으로 전체 파이프라인·위젯·테스트를 먼저 완성하고, 키 수령 시 토스 클라이언트 **어댑터만** 실 호출로 교체한다. fixture와 실 응답은 동일 zod 스키마를 통과해야 한다.
- **시스템 Notion 쓰기 범위**: `종목 DB.현재가` 1개 필드(+선택: `매매일지` 토스 체결행). 그 외 필드는 절대 쓰지 않는다.

## 3. 아키텍처 — 두 데이터 경로 (분리가 핵심)

```
[브라우저: React 위젯]  ──REST/SSE──>  [Node+Express 백엔드]  ──>  [토스 Open API]
                                              │                  └─>  [Notion API]
                                              ├ API 레이어(단기 캐시 TTL 3~5초)
                                              ├ 토스 클라이언트(OAuth 토큰관리·시세·보유·체결·캔들)
                                              ├ overlay-calc(core, 순수함수)
                                              ├ Notion 동기화(현재가 upsert + 선택 체결 임포트)
                                              ├ 스케줄러(node-cron, 장중 1~5분)
                                              └ 스토어(SQLite: 스냅샷·임포트 커서·동기화 상태)
```

| 경로 | 트리거 | 빈도 | 저장 |
|------|--------|------|------|
| **온디맨드 시세** | 위젯 요청 | 수 초 | 단기 캐시(TTL 3~5초) |
| **주기 동기화** | node-cron(장중) | 1~5분 | SQLite + Notion 쓰기 |

두 경로는 **같은 시세 캐시를 공유**한다. 화면은 거의 실시간, Notion 현재가는 배치라 레이트리밋 안전.

  │     ├── /notion-sync     # 현재가 upsert·티커 매칭·체결 임포트·중복방지·소유권
  │     └── /overlay-calc    # 라이브 평가손익·ETF예상가·대조 사양 + 테스트

## 5. 도메인 불변식 (위반 금지)

1. **현재가는 시스템이 유일하게 주입(쓰기)한다.** 평단가·평가금액·평가손익·수익률·보유수량은 **Notion 수식/롤업이 계산** — 시스템은 **읽기/대조만**, 역방향 쓰기 금지.
2. **필드 소유권**: `종목 DB`에서 시스템 쓰기 가능 필드는 `현재가`(+선택 `데이터 플래그`)뿐. 사용자/수식 소유 필드는 보존한다.
3. **티커/코드를 매칭 키**로 Notion page를 찾아 upsert한다.
4. **체결 임포트(선택)**: `매매일지`에 **증권사=토스증권** 행만 생성. **중복키 = 체결ID/시각 + 종목 + 수량 + 단가**. 자동 생성행도 사용자 편집은 보존.
5. **멀티 증권사 경계**: 토스 API는 토스증권 계좌만 조회. 미래에셋·키움 거래는 수동 입력 유지(현재가 주입은 종목 단위라 증권사 무관).
6. **ETF 예상가**: 구성종목·비중은 **사용자가 Notion에 수동 등록**. `ETF 예상등락률 = Σ(구성 등락률 × 비중)`, `예상가 ≈ 기준가 × (1+예상등락률)` — "iNAV 근사·참고용" 고지 필수.
7. **`매매일지.실현손익` 수식은 과대계상 가능성이 있으나(설계서 §5.6) 시스템은 건드리지 않는다.** 보정은 사용자 몫.
8. **장 운영시간**: 장중에만 폴링/동기화, 장외엔 종가 유지(`MARKET_CLOSED`).

## 6. 코딩 규약

- **시크릿(C1)**: `client_secret`·access_token은 **서버에만**. 브라우저 노출 금지, 프론트는 자체 백엔드만 호출. 시크릿은 환경변수로.
- **TypeScript**: strict. 외부 경계(토스 응답·Notion 응답·API 입력)는 **zod로 파싱 후** 사용. `any` 지양.
- **에러 처리**: 외부 호출은 재시도 3회(지수 백오프). 실패 시 **스킵 + 로그 + 직전 캐시 유지**, 사이클 전체를 죽이지 않는다. Notion 쓰기는 레코드 단위 재시도 후 실패분을 `sync_failures.json` 큐로.
- **레이트리밋**: 토스 = 순차 + 백오프(C4). Notion = **~3 req/s 준수**, 틱을 그대로 쓰지 말고 주기 배치(C5).
- **토큰(C3)**: `expires_in≈3600` → 서버 캐싱·자동 갱신(만료 60초 전 갱신).
- **로깅**: 사이클 상태를 `LIVE / DEGRADED / MARKET_CLOSED`로 기록. 시크릿·토큰은 로그에 남기지 않는다.
- **순수성**: `packages/core`(overlay-calc)는 부수효과 없는 순수함수. NaN 금지.

## 7. 검증 / 테스트 정책

- `packages/core`의 overlay-calc(라이브 평가손익·ETF 예상가·평단가 대조)는 **Vitest 단위테스트 필수**.
- 외부 응답은 zod 스키마 검증을 통과해야 진행(시세: 가격>0·신선도, 보유: 종목코드/수량/통화).
- 시세 동기화 성공 임계: 대상의 **≥90%** 확보. 미만이면 사이클 스킵.
- Notion 쓰기 후 검증: 쓰기 성공(200) + 소유권 필드 미변경 + 체결 중복 0. 보유수량/현재가 급변 시 사람 검토 플래그.
- Mock/Fixture와 실 토스 응답은 **동일 스키마**로 검증 — 어댑터 교체 시 테스트가 회귀 가드 역할.

## 8. 빌드 순서 (버티컬 슬라이스)

1. 데이터 파이프라인: 토스(fixture) 토큰→시세→보유 + 캐시(1종목).
2. **Notion 현재가 주입**: `종목 DB` 1종목 티커 매칭 upsert → 평가금액/손익 수식 갱신 확인.
3. overlay-calc + 단위테스트(라이브 평가손익·대조).
4. (선택) 토스 체결 → `매매일지` 임포트 + 중복방지.
5. 위젯: 관심종목 테이블 → 캔들차트 → ETF 히트맵 (DESIGN.md 토큰 적용).
6. 종단 테스트: (키 수령 후) 실계좌 1~2종목 전체 흐름.

## 9. 프론트 UI 표준

`apps/web` 위젯은 [DESIGN.md](DESIGN.md)의 Notion 디자인 시스템을 **표준으로 따른다**:
- 색은 CSS 변수로(하드코딩 금지). Notion 따뜻한 중성 팔레트, Notion Blue(`#5c5c5c`)는 CTA/링크에만.
- 위스퍼 보더 `1px solid rgba(0,0,0,0.1)`, 다층 저투명 섀도, NotionInter/Inter 타이포 스케일.
- **데이터 밀집 레이아웃**(설계서 위젯 성격) — DESIGN.md §5의 compact/dense 원칙 적용. 한 화면에 더 많은 종목.
- 등락은 적/녹 의미색(설계서 ETF 히트맵). 다크모드 토큰은 DESIGN.md 표 준수.
- **이모지 금지** — 상태/등락은 컬러 닷·SVG 아이콘으로.
