# 토스증권 × Notion 주식 위젯 사이트 — 통합 설계서

> 이 문서는 Claude Code로 구현할 때 참조하는 **계획서**다.
> CLAUDE.md / SKILL.md / 모듈의 **상세 내용은 여기서 쓰지 않는다**. 구조·역할·계약·정책까지만 정의한다.
> 성격: **런타임 LLM이 없는 결정론적 웹앱 + 데이터 동기화 파이프라인**. 여기서 "에이전트"는 *빌더로서의 Claude Code*를 가리킨다.
> **연동 대상: 사용자의 기존 Notion 템플릿(증권사 계좌 / 매매일지 DB / 종목 DB)** — §부록 B.

## 1. 작업 컨텍스트

### 1.1 배경
- 토스증권 Open API 키 신청 후 수령 대기. 수령 시 토스증권 계좌의 시세·보유내역·체결내역에 프로그래밍 접근 가능.
- **사용자는 이미 Notion 매매일지 템플릿(거래원장 기반 자동계산)을 운용 중.** 이 시스템은 그 템플릿을 대체하지 않고 **현재가를 주입**해 살아 움직이게 만든다.
- 기존 정적 Notion 위젯과 달리, 시크릿 보관·폴링 때문에 **상시 가동 백엔드가 필요**하다.

### 1.2 목적
관심·보유·ETF 구성을 한 화면에서 보고(시세·차트·히트맵), **기존 Notion `종목 DB`에 현재가를 자동 주입**해 평가금액·평가손익·수익률이 실시간에 가깝게 갱신되도록 한다. (선택) 토스 체결내역을 `매매일지`에 자동 기록한다.

### 1.3 범위
**포함**
- 폴링 기반 실시간 시세 위젯 / 관심종목 모음(테이블) / 차트(캔들) / ETF 예상가 히트맵
- 토스 시세 → `종목 DB.현재가` 주기 동기화
- (선택) 토스증권 체결내역 → `매매일지 DB` 자동 임포트(중복 방지)

**제외**
- 자동 주문/매매(토스 Order 미사용 — 조회 전용)
- 런타임 LLM, 다중 사용자 SaaS화
- 평단가·평가손익 **재계산 후 Notion에 쓰기**(이미 Notion 수식이 담당)
- 미래에셋·키움 등 **타 증권사 거래의 자동 수집**(토스 API 범위 밖 → 수동)

### 1.4 입력 / 출력

| 구분 | 내용 |
|------|------|
| **입력(외부)** | 토스 시세(현재가·캔들 OHLCV), 토스증권 계좌 보유내역·체결내역, 토스 장 운영시간 |
| **입력(Notion 읽기)** | `종목 DB`의 티커/코드·보유수량·평단가(위젯 실시간 오버레이용), `매매일지` 키(중복판정용), ETF 구성종목·비중(§5.4) |
| **출력(주)** | **`종목 DB.현재가` 갱신** → Notion 수식이 평가금액·평가손익·수익률 연쇄 계산 |
| **출력(선택)** | `매매일지 DB`에 토스증권 체결 신규행(증권사=토스증권, 중복 방지) |
| **출력(부)** | 위젯 화면(시세·차트·히트맵), `/output` 캐시 스냅샷(JSON)·동기화 로그 |

### 1.5 제약조건
- **C1. 시크릿 서버 보관**: `client_secret`·access_token은 브라우저 노출 금지 → 프론트는 자체 백엔드만 호출.
- **C2. WebSocket 없음(2026-06)**: 실시간 = REST 폴링(최소 ~1초) + 단기 캐시.
- **C3. 토큰 수명**: `expires_in≈3600` → 서버 캐싱·자동 갱신.
- **C4. 토스 레이트리밋**: 순차 + 백오프.
- **C5. Notion 레이트리밋(~3 req/s)**: 틱을 그대로 쓰지 않고 **주기 배치**.
- **C6. 장 운영시간**: 장중에만 폴링, 장외엔 종가 유지.
- **C7. 멀티 증권사 경계**: 토스 API는 **토스증권 계좌만** 조회 → 타 증권사 거래는 수동.
- **C8. 충돌 방지**: 시스템 쓰기 범위를 `현재가`(+선택 체결행)로 최소화해 사용자 입력과 충돌 자체를 줄인다.

### 1.6 용어 정의
- **평단가**: 보유분 매입 원가 평균. **본 시스템에서는 Notion이 `매매일지`로부터 수식 계산**(시스템은 사용/대조만).
- **iNAV / 괴리율**: ETF 이론가(구성종목 가중) / 실제가와의 차이. 히트맵은 참고용.
- **upsert**: 키 매칭 갱신/생성.
- **현재가 주입**: 시스템이 외부 시세를 `종목 DB.현재가`에 쓰는 동작(이 시스템의 핵심 쓰기).
- **체결 임포트**: 토스 체결내역을 `매매일지` 행으로 적재(선택).

## 2. 시스템 아키텍처

### 2.1 컴포넌트 개요

```
[브라우저: React 위젯들]
        │  (자체 백엔드 REST/SSE만 호출)
        ▼
[Node + Express 백엔드]
   ├── API 레이어        : 위젯에 라이브 시세/캔들/히트맵 제공 (단기 캐시)
   ├── 토스 클라이언트    : OAuth 토큰관리 + 시세/보유/체결/캔들 조회
   ├── overlay-calc(core): 라이브 평가손익 오버레이 + 토스/노션 평단가 대조(경량)
   ├── Notion 동기화     : 종목DB '현재가' upsert (+선택: 매매일지 체결 임포트)
   ├── 스케줄러(node-cron): 장중 주기 트리거
   └── 캐시/스토어(SQLite): 시세 스냅샷·체결 임포트 커서·동기화 상태
        │                         ▲
        ▼                         │
   [토스증권 Open API]        [Notion API: 증권사계좌/매매일지/종목 DB]
```

### 2.2 두 가지 데이터 경로 (분리가 핵심)

| 경로 | 트리거 | 목적 | 빈도 | 저장 |
|------|--------|------|------|------|
| **온디맨드 시세** | 위젯 요청 | 화면 실시간성(시세·차트·히트맵·라이브 평가손익) | 위젯 열림 시 수 초 | 단기 캐시(TTL 3~5초) |
| **주기 동기화** | node-cron(장중 N분) | `종목DB.현재가` 갱신 → Notion 수식 연쇄 / (선택)체결 임포트 | 1~5분 | SQLite + Notion 쓰기 |

> 화면은 백엔드에서 라이브 시세를 직접 받아 "거의 실시간", Notion `현재가`는 1~5분 배치로 갱신되어 레이트리밋 안전. 둘 다 같은 시세 캐시를 공유.

### 2.3 상태(스토어)
- 토스 access_token(+만료), 종목별 최신 시세 스냅샷
- **체결 임포트 커서**(마지막으로 임포트한 체결 시각/ID — 중복 방지)
- 동기화 상태(마지막 성공시각·실패 큐·티커→Notion page_id 매핑)

## 3. 워크플로우(동기화 파이프라인) — 단계별 정의

각 단계에 **성공 기준 / 검증 / 실패 처리**를 명시. 모든 판단은 코드 규칙, 런타임 LLM 판단 **없음**.

### S0. 토큰 확보·갱신
- **성공**: 유효 토큰 확보. **검증**: 만료시각 > 현재+60초.
- **실패**: 재시도 3회(지수 백오프) → 실패 시 사이클 **스킵+로그**(이전 캐시 유지).

### S1. 토스증권 보유·체결 조회
- **처리**: 토스증권 계좌 보유내역(종목·수량·평균매입가) + (선택)신규 체결내역(커서 이후).
- **성공**: 응답 스키마 충족. **검증**: 스키마 검증(종목코드/수량/통화).
- **실패**: 재시도 3회 → 직전 스냅샷 유지 + **스킵+로그**.
- **주의**: 토스증권 계좌만. 타 증권사 데이터는 여기서 들어오지 않음.

### S2. 시세 조회 (종목DB 티커 ∪ 관심 ∪ ETF구성)
- **처리**: 대상 종목 현재가/등락률. **순차+백오프**.
- **성공**: 대상의 ≥90% 시세 확보. **검증**: 규칙 기반(가격>0, 신선도).
- **실패**: 개별 실패는 **스킵+로그**(셀 stale), 전체 실패는 사이클 스킵.

### S3. 경량 계산 (overlay-calc, core)
- **처리**: 위젯용 **라이브 평가손익 오버레이** = `(라이브현재가 − 평단가) × 보유수량`(평단가·수량은 Notion 값). ETF 예상가(§5.4). 토스 평균매입가 ↔ Notion 평단가 **대조 플래그**(불일치 시).
- **성공**: 계산 완료, NaN 없음. **검증**: 스키마+규칙. 단위테스트.
- **실패**: 해당 종목 **스킵+로그**.
- **주의**: **평단가·평가금액 자체는 계산/쓰기하지 않는다(Notion 수식 담당).**

### S4. Notion 쓰기 (배치)
- **처리(필수)**: `종목DB.현재가` upsert — **티커/코드로 page 매칭**, ~3 req/s 준수.
- **처리(선택)**: `매매일지`에 토스 체결 신규행 생성(증권사=토스증권, **중복키=체결ID/시각+종목+수량+단가**).
- **성공**: 현재가 전부 반영, 사용자 소유 필드 미변경, 체결 중복 0.
- **검증**: 규칙 기반(쓰기 200, 소유권 보존) + 보유수량/현재가 급변 시 **사람 검토 플래그**.
- **실패**: 레코드 단위 재시도 → 실패분 **실패 큐+로그**, 다음 사이클 재시도.

### S5. 프론트 제공
- 캐시 스냅샷을 REST(JSON)/SSE로. 신선도·TTL 메타 포함. 실패 시 마지막 캐시 + "지연" 배지.

### 상태 전이
- 장외/휴장 → S2~S4 중단, 종가 유지(`MARKET_CLOSED`).
- 토큰/보유 연속 실패 → `DEGRADED`(stale 표시). 정상 → `LIVE`.

## 4. 판단과 코드의 역할 분리

| 에이전트(=Claude Code, **빌드 시에만**) | 런타임 코드(**항상**) |
|---|---|
| 설계서 해석, 모듈 골격·스키마 생성 | 토스/노션 호출, OAuth 토큰 관리 |
| 테스트 케이스 초안, 엣지케이스 문서화 | 시세 폴링·캐시·백오프·레이트리밋 |
| 티커 매칭·중복키 규칙 구체화 보조 | `종목DB.현재가` upsert, 체결 임포트(중복방지) |
| — | overlay-calc(라이브 평가손익·ETF예상가·대조 플래그) |
| — | 장 운영시간 판정·스케줄링 |

> **런타임 LLM 판단 영역: 없음.** 평단가·손익은 Notion 수식. 시스템은 현재가 주입 + 시세 제공 + (선택)체결 임포트만.

## 5. 핵심 도메인 규칙

### 5.1 하이브리드 데이터 모델 원칙
- **Notion이 1차 원장**(매매일지)이자 계산기(수식). 토스 = 시세·토스증권 거래의 자동 공급원.
- 동기화 방향: 토스 시세 → `종목DB.현재가`(쓰기). 토스 체결 → `매매일지`(선택 쓰기). **평단가·손익은 역방향으로 쓰지 않음.**

### 5.2 평단가 — Notion 위임(시스템은 대조만)
- `종목DB.평단가`·`보유수량(순)`·`총보유금액`은 **Notion이 `매매일지`에서 수식/롤업으로 계산**. 시스템은 **읽기만**.
- 시스템은 토스 보유내역의 평균매입가를 가져와 **Notion 평단가와 대조**, 불일치 시 `데이터 플래그`로 표시(선택). 자동 덮어쓰기 없음.
- 위젯의 라이브 평가손익은 `(라이브현재가 − Notion평단가) × Notion보유수량`으로 즉석 계산(저장 안 함).

### 5.3 필드 소유권 규칙 (실제 3-DB 기준)

**종목 DB**
| 필드 | 소유/생성 | 동기화 동작 |
|------|----------|------------|
| 현재가 | **시스템** | 매 사이클 **덮어쓰기(유일 주입)** |
| 평가금액·평가손익·수익률·보유기간 | Notion 수식 | 시스템 X(현재가로 자동 연쇄) |
| 보유수량(순)·평단가·총보유금액 | Notion 롤업/수식(매매일지 파생) | 시스템 X(대조만) |
| 종목명·티커/코드·현재 보유·메모 | 사용자 | 시스템 X(티커는 매칭 키로 읽기) |
| 데이터 플래그(대조 불일치 등) | 시스템(선택) | 덮어쓰기 |

**매매일지 DB**
| 필드 | 소유/생성 | 동기화 동작 |
|------|----------|------------|
| 행(날짜·종목·증권사·구분·수량·단가) | **하이브리드** | 토스증권 체결 **자동 임포트(중복키)** + 수동 입력. 자동행도 사용자 편집 보존 |
| 회고 | 사용자 | 시스템 X |

**증권사 계좌 DB**: 전부 사용자/Notion 롤업 — 시스템 X.

### 5.4 ETF 예상가 히트맵
- **구성종목·비중 출처(미해결)**: 템플릿엔 ETF 구성내역이 없음. 기본안 = **사용자가 Notion에 구성종목·비중 등록**(예: 0167A0 SOL AI반도체TOP2플러스의 구성) → 각 구성종목 현재가 폴링.
- 계산: `ETF 예상등락률 = Σ(구성종목 등락률 × 비중)`, `예상가 ≈ 기준가 × (1+예상등락률)`(iNAV 근사).
- 히트맵: 셀=구성종목, 색=등락률(적/녹), 크기=비중(treemap). **"iNAV 근사·참고용" 고지**.

### 5.5 멀티 증권사 규칙
- 토스 API는 **토스증권 계좌만**. 따라서 체결 자동 임포트는 `매매일지`의 **증권사=토스증권 행만** 생성.
- 미래에셋·키움 거래는 **수동 입력 유지**. 시스템은 이 행을 읽지도 쓰지도 않음(현재가 주입은 종목 단위라 증권사 무관하게 모든 보유 종목에 적용).

### 5.6 (선택) 실현손익 정의 점검 — 정확도 권장사항
- 현재 `매매일지.실현손익`이 매도대금(거래금액)과 동일해 보임 → 진짜 실현손익(매도대금 − 평단가×수량)과 차이.
- 시스템은 이 컬럼을 건드리지 않으나, `증권사 계좌.총실현손익` 롤업 과대계상 가능 → **수식 보정 권장**(integration 전 사용자 결정).

## 6. 백엔드 실행 환경 — 추천 및 근거

### 6.1 요구사항
상시 가동(장중 주기 동기화) · 시크릿 서버 보관 · OAuth 토큰 캐싱 · Notion 배치 쓰기 · 기존 Node+Express 일치 · 저비용·저운영.

### 6.2 추천: **Railway(또는 Render) 단일 Node+Express 서비스**
- 하나의 서비스에 **API + node-cron** 동거. 데이터 **SQLite(better-sqlite3)** 또는 Railway Postgres. 프론트(Vite 정적)는 동일 서비스 서빙 또는 Cloudflare Pages/Vercel.
- 이유: 기존 스택 그대로, 단일 배포, 상시+cron, 환경변수 시크릿, 저비용.

### 6.3 대안 비교
| 옵션 | 적합성 | 비용 | 난이도 | 제약 |
|------|--------|------|--------|------|
| **Railway/Render(추천)** | 높음 | 낮음 | 낮음 | 상시 컨테이너 소액 비용 |
| Cloudflare Workers+Cron+D1 | 중 | 매우 낮음 | 중 | Express 미사용·런타임 상이, cron 분 단위 |
| VPS/홈서버 | 높음 | 0~낮음 | 중~높 | 운영·보안·가동률 직접 책임 |

> 온디맨드 시세(요청 시 조회+단기 캐시) + cron 배치 동기화 분리로 상시 1초 폴링 불필요 → Railway 단일 서비스로 충분.

## 7. 구현 스펙 (구조·역할 수준만)

### 7.1 폴더 구조 (npm workspaces 모노레포)
```
/toss-notion-widget
  ├── package.json
  ├── CLAUDE.md
  ├── /.claude
  │   └── /skills
  │       ├── /toss-api-ref      # SKILL.md + /scripts(토큰·시세·보유·체결·캔들) + /references(엔드포인트·레이트리밋)
  │       ├── /notion-sync       # SKILL.md + /scripts(현재가 upsert·티커 매칭·체결 임포트·중복방지)
  │       └── /overlay-calc      # SKILL.md (라이브 평가손익·ETF예상가·대조 사양 + 테스트)
  ├── /apps
  │   ├── /server                # Express API + cron + 토스/노션 클라이언트
  │   └── /web                    # React+Vite 위젯(시세/테이블/차트/히트맵)
  ├── /packages
  │   └── /core                   # overlay-calc 순수함수 + 공유 타입/스키마(zod)
  ├── /output                     # 캐시 스냅샷·임포트 커서·실패 큐·로그
  └── /docs
```
> `/.claude/skills`는 **런타임 스킬이 아니라 Claude Code 빌드 참조 묶음**. 런타임 서브에이전트 없음.

### 7.2 CLAUDE.md 핵심 섹션(제목만)
목적·범위/비범위 · 아키텍처(두 경로) · 빌드 순서(§7.7) · 코딩 규약(시크릿·에러·로깅) · 도메인 불변식(§5) · 검증·테스트 정책 · 참조 스킬 인덱스.

### 7.3 에이전트 구조
- 빌드 시: 단일 메인(CLAUDE.md) 오케스트레이터. 서브에이전트 불필요.
- 런타임: 에이전트 없음.

### 7.4 스킬(빌드 참조) 목록·역할·트리거
| 스킬 | 역할 | 트리거 |
|------|------|--------|
| `toss-api-ref` | OAuth/시세/보유/체결/캔들 래퍼·엔드포인트맵·레이트리밋 | 토스 연동 코드 작업 시 |
| `notion-sync` | 현재가 upsert, **티커→page 매칭**, 체결 임포트·**중복방지**, 소유권 규칙, 배치 | Notion 쓰기 작업 시 |
| `overlay-calc` | 라이브 평가손익·ETF 예상가·평단가 대조 사양 + 단위테스트 | core 구현·검증 시 |

### 7.5 모듈(런타임) 책임
| 모듈 | 책임 |
|------|------|
| `server/toss/*` | 토큰, 시세/보유/체결/캔들 조회, 백오프 |
| `server/notion/*` | 현재가 upsert, 티커 매칭, 체결 임포트(중복키), 소유권 준수, 배치 |
| `server/scheduler` | node-cron, 장 운영시간 판정, 사이클 상태머신, 임포트 커서 |
| `server/api` | 위젯 REST/SSE, 단기 캐시 |
| `packages/core` | overlay-calc(순수), 타입·스키마, 검증 |
| `web/widgets/*` | 시세 테이블·차트(캔들)·ETF 히트맵·관심종목 모음 |

### 7.6 데이터 전달
- 중간 산출물 `/output/`(예: `holdings_snapshot.json`, `import_cursor.json`, `sync_failures.json`).
- 위젯↔백엔드: 작은 JSON 인라인, 캔들은 캐시 파일·엔드포인트.

### 7.7 빌드 순서 (버티컬 슬라이스)
1. 데이터 파이프라인: 토스 토큰→시세→보유 조회 + 캐시(1종목).
2. **Notion 현재가 주입**: `종목DB` 1종목 티커 매칭 upsert → 평가금액/손익 수식 갱신 확인.
3. overlay-calc + 단위테스트(라이브 평가손익·대조).
4. (선택) 토스 체결 → `매매일지` 임포트 + 중복방지.
5. 위젯: 관심종목 테이블 → 차트 → ETF 히트맵.
6. 종단 테스트: 실계좌 1~2종목 전체 흐름.

### 7.8 주요 산출물 형식
- **Notion `종목DB.현재가` 갱신**(주) → 수식 연쇄. (선택)`매매일지` 임포트 행.
- `/output/*.json`(시세·보유 스냅샷·임포트 커서·실패 큐), 사이클 로그(`LIVE/DEGRADED/MARKET_CLOSED`).
